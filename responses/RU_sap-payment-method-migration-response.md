# SAP Способы оплаты — Поэтапная стратегия миграции форматов

---

## Краткое резюме

Данный документ представляет безопасную для продуктива поэтапную стратегию миграции всех способов оплаты в нескольких балансовых единицах с устаревших платёжных форматов (классические программы RFFO* или устаревшие деревья форматов PMW) на современные форматы ISO 20022 (pain.001.001.09 / CGI XML / пользовательские деревья форматов PMW).

**Рекомендуемый основной подход: Дополнения к способам оплаты (Zahlwegergänzung).**

Механизм дополнений (поле `UZAWE` в `LFB1`, конфигурация в `T042D`) позволяет назначить новый платёжный формат 2-символьному расширению существующего кода способа оплаты. Это обеспечивает гранулярность на уровне поставщика, не требует новых символов способов оплаты, сохраняет всю существующую конфигурацию FBZP и позволяет мгновенно откатиться, просто очистив дополнение в мастер-данных поставщика. Вся миграция может выполняться в рамках одного платёжного прогона F110 — поставщики со старым и новым форматами сосуществуют прозрачно.

**Ориентировочные сроки:** 12-16 недель для 10 способов оплаты в нескольких балансовых единицах.  
**Ориентировочные трудозатраты:** 45-60 человеко-дней для команды из 2 SAP FI консультантов.

### Ключевые сроки ISO 20022

| Срок | Требование |
|---|---|
| Ноябрь 2025 | SWIFT прекращает поддержку формата MT (MT940, MT942, MT100); требуется миграция на сообщения camt/pain |
| 15 ноября 2026 | Мандат SEPA: неструктурированные почтовые адреса более не принимаются; требуется pain.001 версии 2019 (pain.001.001.09) |
| Постоянно | Расширяется внедрение LEI (Legal Entity Identifier) и UETR (Unique End-to-End Transaction Reference) |

Эти сроки означают, что организации, по-прежнему использующие устаревшие форматы, сталкиваются с жёсткими датами отсечения со стороны банков и клиринговых организаций. Поэтапный подход к миграции не просто желателен — это единственный ответственный путь с учётом сжатых сроков и серьёзности нарушений платёжного процесса в случае сбоя нового формата.

## Часть A — Рекомендуемый подход

### A.1 Оценка механизмов миграции

#### A.1.1 Дополнения к способам оплаты (Zahlwegergänzung)

**Вердикт: РЕКОМЕНДУЕТСЯ — Основной механизм.**

Дополнения к способам оплаты расширяют базовый способ оплаты (один символ, например `T`) 2-символьным суффиксом (например, `01`). Дополнение хранится в записи мастер-данных поставщика (`LFB1-UZAWE`) и считывается при обработке F110. Если в `T042D` / `TFPM042FZ` существует специфическая конфигурация формата для дополнения, формат PMW, назначенный дополнению, переопределяет формат базового способа оплаты.

**Почему подходит для данного сценария:**

1. **Никакого исчерпания символов.** Код базового способа оплаты остаётся неизменным. Дополнения используют отдельное 2-символьное поле с до 100 комбинаций (00-99) на способ оплаты, что значительно превышает потребности миграции.
2. **Гранулярность на уровне поставщика.** Каждый поставщик может быть индивидуально назначен на новый формат (дополнение `01`) или оставлен на старом формате (без дополнения / пустой `UZAWE`). Это обеспечивает точное развёртывание волнами: 5 поставщиков, 50 поставщиков, 500 поставщиков.
3. **Стандартная функциональность SAP.** Никакого пользовательского кода. Настраивается полностью через `FBZP`, `OBPM` и транзакции мастер-данных поставщиков. Поддерживается SAP начиная с R/3 4.6C.
4. **Прозрачно для пользователей F110.** Параметры платёжного прогона, варианты выборки и рабочие процессы пользователей не изменяются. Один и тот же прогон F110 обрабатывает и поставщиков со старым, и с новым форматом — формат разрешается для каждой строки `REGUH` на основе дополнения поставщика.
5. **Мгновенный откат.** Очистка `LFB1-UZAWE` (установка в пустое значение) для любого поставщика немедленно возвращает этого поставщика к формату базового способа оплаты. Перенастройка FBZP не требуется.
6. **Прослеживаемость.** Изменение поля `UZAWE` регистрируется в истории изменений мастер-данных поставщика (таблица `CDPOS` / `CDHDR`, класс объекта `KRED`). В сочетании с данными `REGUH` существует полный аудиторский след, показывающий, какой поставщик использовал какой формат в каком платёжном прогоне.

**Ограничения, которые нужно учесть:**

- Требует изменений мастер-данных поставщиков (нивелируется массовым обновлением через LSMW / BDC — см. раздел A.2.4).
- Поле `UZAWE` может уже использоваться в других целях (проверьте текущее использование через `SE16` по `LFB1` до начала; см. риск D.4).

#### A.1.2 Переопределение формата на уровне домашнего банка / идентификатора счёта

**Вердикт: НЕ РЕКОМЕНДУЕТСЯ в качестве основного механизма.**

Этот подход создаёт параллельный идентификатор счёта под тем же домашним банком, назначает ему новый формат и корректирует приоритеты определения банка в `FBZP` → Определение банка (`T042E`).

**Почему не подходит:**

- **Нет гранулярности на уровне поставщика.** Определение банка работает на уровне способа оплаты + балансовая единица, а не для каждого поставщика. Все поставщики, использующие одинаковый способ оплаты и балансовую единицу, маршрутизируются через одинаковый приоритет домашнего банка. Нельзя выборочно направить 10 пилотных поставщиков через новый идентификатор счёта, оставив других на старом, в рамках одного прогона F110.
- **Сложная логика приоритетов.** Манипулирование приоритетами определения банка для частичной маршрутизации нестабильно и чревато ошибками, особенно при наличии правил приоритетности по суммам.
- **Ограниченный откат.** Откат изменений определения банка требует тщательного восстановления приоритетов и несёт риск ошибок маршрутизации платежей.

**Когда полезно:** В качестве дополнительного механизма, когда новый формат также требует другого банковского счёта (например, новый банк-корреспондент), что ортогонально самой миграции формата.

#### A.1.3 Динамическое переключение форматов через BAdI

**Вердикт: РЕКОМЕНДУЕТСЯ — Дополнительное / опциональное улучшение, не основной механизм.**

Реализация BAdI `FI_PAYMEDIUM` (или `BADI_PAYMETHOD_SELECTION` для логики на этапе предложения) позволяет выбирать формат во время выполнения на основе произвольных критериев: номер поставщика, группа поставщиков, диапазон дат, порог суммы или пользовательская Z-таблица со списком пилотных поставщиков.

**Почему привлекательно, но не является основным:**

- **Максимальная гибкость.** Любая логика выбора может быть запрограммирована: переключение по дате, волны по группам поставщиков, случайная выборка по процентам для A/B-тестирования.
- **Не требует изменений мастер-данных.** Список пилотных поставщиков может находиться в Z-таблице, полностью избегая обновлений мастер-данных поставщиков.
- **Полное логирование.** Пользовательский код может записывать детальные журналы миграции (поставщик, способ оплаты, использованный формат, дата, идентификатор прогона).

**Почему не должен быть основным:**

- **Пользовательский код в критическом пути платежей.** Любая ошибка в логике BAdI может привести к повреждению платёжного файла, дублированию платежей или сбоям платёжного прогона. Нагрузка на аудит и тестирование пользовательского кода в обработке `SAPFPAYM` значительно выше, чем для стандартной конфигурации.
- **Затраты на поддержку.** Реализация BAdI должна поддерживаться, транспортироваться по средам, регрессионно тестироваться с каждым пакетом поддержки SAP и в конечном итоге выводиться из эксплуатации. Для временного инструмента миграции это несоразмерные усилия.
- **Не «стандартный SAP».** Аудиторы и банковские операционные группы могут потребовать дополнительного согласования пользовательского кода в платёжной цепочке.

**Рекомендуемое гибридное использование:** Реализовать лёгкий BAdI `FI_PAYMEDIUM`, который только логирует выбранный формат для каждого поставщика в каждом прогоне (только чтение, без логики переключения форматов). Это обеспечивает аудиторский след без риска изменения платёжного поведения. Фактическая маршрутизация форматов обрабатывается механизмом дополнений.

#### A.1.4 Временный дублирующий способ оплаты

**Вердикт: НЕ РЕКОМЕНДУЕТСЯ для данного сценария.**

Создание временного способа оплаты (например, `X` как копия `T`) с назначенным новым форматом, затем перевод пилотных поставщиков на `X` в их мастер-данных.

**Почему не подходит:**

- **Исчерпание символов.** При наличии уже 10 задействованных способов оплаты дублирование всех 10 требует 10 дополнительных символов (20 итого). Если другие символы уже заняты неактивными или страновыми способами оплаты, это может оказаться невозможным.
- **Дублирование конфигурации.** Каждая настройка FBZP, запись определения банка, назначение домашнего банка, назначение формы, значение объекта полномочий и вариант выборки должны быть продублированы для каждого временного способа оплаты. Для нескольких балансовых единиц это умножается дополнительно.
- **Перегрузка мастер-данных.** Поставщиков нужно изменять дважды: сначала на временный способ оплаты, затем обратно на исходный (с новым форматом) после миграции. Двойные трудозатраты на массовое обновление, двойной риск ошибок.
- **Влияние на F110.** Варианты выборки и параметры платёжного прогона могут потребовать корректировки для включения новых кодов способов оплаты. Это изменяет рабочие процессы пользователей.

**Когда полезно:** Только когда дополнения недоступны (например, поле `UZAWE` заблокировано другим процессом), И количество мигрируемых способов оплаты очень мало (1-2), И имеются достаточные свободные символы.

#### A.1.5 Параллельные прогоны F110

**Вердикт: НЕ РЕКОМЕНДУЕТСЯ в качестве основного механизма.**

Выполнение двух отдельных платёжных прогонов F110 в день платежа — одного для поставщиков со старым форматом, другого для поставщиков с новым форматом — с использованием разных критериев выборки или способов оплаты.

**Почему не подходит:**

- **Операционная сложность.** Удваивает количество платёжных прогонов. Планирование прогонов, передача банковских файлов, сверка и управление денежными средствами — всё усложняется.
- **Изменения рабочих процессов пользователей.** Операторы F110 должны управлять двумя прогонами вместо одного, с логикой выборки поставщиков, которую нужно поддерживать и обновлять по мере расширения пилотной группы.
- **Риск дублирования платежей.** Если выборка поставщиков между двумя прогонами не является идеально взаимоисключающей, поставщик может быть обработан в обоих прогонах, что приведёт к дублированию платежей.
- **Фрагментация кассовой позиции.** Два отдельных платёжных прогона генерируют два отдельных банковских файла, что потенциально усложняет прогнозирование денежных средств и банковскую выверку.

**Когда полезно:** Для первоначального дымового тестирования в QAS (выполнить параллельный F110 с 2-3 поставщиками на новом формате для проверки вывода DME до запуска дополнений в PRD).

#### A.1.6 Другие рассмотренные механизмы

**Переключение варианта выборки PMW (`OBPM4`):**
Варианты выборки управляют специфическими для формата параметрами (например, группировка, критерии домашнего банка), но не обеспечивают маршрутизацию формата на уровне поставщика. Не подходят в качестве основного рычага миграции.

**Специфические для клиента/поставщика форматы через корреспонденцию (BTE 00001820):**
Бизнес-транзакционное событие 00001820 может влиять на обработку платёжного носителя, но предназначено для настройки корреспонденции, а не для переключения форматов. Не рекомендуется.

**Определение вывода / условный формат (только S/4HANA):**
В S/4HANA управление выводом теоретически может маршрутизировать разные форматы на основе условий. Однако это не является зрелым решением для форматов платёжных носителей по состоянию на S/4HANA 2023 и добавляет излишнюю сложность.

### A.2 Детальное описание — Подход через дополнения к способам оплаты

#### A.2.1 Шаги настройки FBZP

Следующая конфигурация должна быть выполнена для каждого способа оплаты (например, `T`) в каждой балансовой единице (например, `1000`). Повторить для всех 10 способов оплаты во всех балансовых единицах.

**Шаг 1: Определить дополнение в способах оплаты по стране**

1. Транзакция: `FBZP`
2. Перейти в: **Способы оплаты по стране** → выбрать страну (например, `DE`)
3. Выбрать способ оплаты (например, `T`)
4. Убедиться, что флажок / индикатор «Дополнение к способу оплаты» включён для этого способа оплаты. Если нет — активировать.
5. Сохранить и отметить запрос на перенос.

> Затронутая таблица: `T042C` — поле `XZAWE` (индикатор допустимости дополнения).

**Шаг 2: Создать конфигурацию дополнения**

1. Транзакция: `FBZP` → **Способы оплаты в балансовой единице**
2. Выбрать балансовую единицу `1000`, способ оплаты `T`
3. Перейти в раздел **Дополнения** (или использовать транзакцию `OBPM`)
4. Создать новое дополнение: `01`
5. В конфигурации дополнения назначить **новый формат PMW**:
   - Поле: «Формат платёжного носителя» → ввести ключ нового формата (например, `SEPA_CT`, `CGI_XML_V3` или ваш пользовательский `Z_ISO20022_CT`)
   - Поле: «Дополнение формата» (в `TFPM042FZ`) → если используется дополнение формата PMW, ввести здесь
6. Назначить соответствующий **вариант выборки** (созданный в `OBPM4`), если новый формат его требует
7. При необходимости назначить новую **форму** для платёжного авизо / сопроводительного листа, если изменение формата требует другого макета
8. Сохранить. Перенести в QAS и PRD.

> Затронутые таблицы: `T042D` (определение дополнения), `TFPM042FZ` (маппинг формата к дополнению), `T042Z` (назначение формата на уровне балансовой единицы для дополнения).

**Пример конфигурации:**

```
FBZP → Способы оплаты в балансовой единице
  Балансовая единица: 1000
  Способ оплаты:      T (Банковский перевод)
  Дополнение:         01
  Формат PMW:         SEPA_CT  (новый формат ISO 20022 pain.001.001.09)
  Вариант выборки:    SEPA_CT_1000_T01 (создан в OBPM4)
  Форма:              F110_SEPA_AVIS (если отличается от базовой)
```

Базовый способ оплаты `T` (без дополнения) сохраняет старое назначение формата:

```
FBZP → Способы оплаты в балансовой единице
  Балансовая единица: 1000
  Способ оплаты:      T (Банковский перевод)
  Дополнение:         (пусто / базовый)
  Формат PMW:         RFFOD__T  (старая классическая программа — без изменений)
```

**Шаг 3: Создать вариант выборки PMW**

1. Транзакция: `OBPM4`
2. Создать новый вариант выборки для нового формата (например, `SEPA_CT_1000_T01`)
3. Установить специфические для формата параметры: домашний банк, критерии группировки платежей, валюта, флаги структурированного адреса
4. Назначить этот вариант дополнению `01` в `FBZP` / `OBPM`

**Шаг 4: Проверить дерево форматов**

1. Транзакция: `DMEE`
2. Открыть целевое дерево форматов (например, `SEPA_CT` или ваше пользовательское дерево)
3. Проверить маппинг всех обязательных полей: IBAN, BIC, блоки структурированного адреса, информация о назначении платежа, UETR
4. Проверить, что версия XML-схемы соответствует требованиям банка (pain.001.001.09 для мандата 2026 года)

#### A.2.2 Трассировка разрешения формата во время выполнения

При выполнении платёжного прогона следующая логика определяет, какой формат используется для каждой строки платежа:

```
F110 (Платёжный прогон)
  │
  ├─ Прогон предложения (SAPFPAYM_SCHEDULE → SAPFPAYM)
  │   ├─ Чтение открытых позиций по поставщику
  │   ├─ Определение способа оплаты из мастер-данных поставщика (LFB1-ZWELS)
  │   ├─ Чтение дополнения к способу оплаты из мастер-данных поставщика (LFB1-UZAWE)
  │   ├─ Запись заголовка REGUH: RZAWE = Способ оплаты, UZAWE = Дополнение
  │   └─ Запись позиций REGUP: строки позиций, сгруппированные под REGUH
  │
  ├─ Платёжный прогон (выполняет платежи, создаёт документы)
  │
  └─ Генерация платёжного носителя (SAPFPAYM → PMW / Классическая программа)
      │
      ├─ Для каждой записи REGUH:
      │   ├─ Чтение REGUH-RZAWE (например, «T») и REGUH-UZAWE (например, «01» или пусто)
      │   │
      │   ├─ ЕСЛИ UZAWE заполнен (например, «01»):
      │   │   ├─ Поиск в T042D для способа оплаты T + дополнение 01
      │   │   ├─ Поиск в TFPM042FZ для специфичного дополнению формата PMW
      │   │   ├─ Найдено: SEPA_CT → использовать новый формат
      │   │   └─ Вызов PMW с деревом форматов SEPA_CT, вариант выборки SEPA_CT_1000_T01
      │   │
      │   └─ ЕСЛИ UZAWE пусто:
      │       ├─ Поиск в T042Z для способа оплаты T в балансовой единице 1000
      │       ├─ Найдено: RFFOD__T → использовать старый формат
      │       └─ Вызов классической программы RFFOD__T (или старого формата PMW)
      │
      └─ Генерация DME-файла(ов) — отдельные файлы на формат
```

**Ключевой вывод:** В рамках одного прогона F110 поставщики с дополнением `01` создают вывод DME через новый формат, а поставщики без дополнения — через старый формат. Генерируются два отдельных DME-файла — по одному на формат. Оба доступны в `FDTA` для проверки и передачи в банк.

#### A.2.3 Назначение поставщиков в пилотную группу

**Обновление отдельного поставщика:**

1. Транзакция: `XK02` (или `FK02` для FI поставщиков, `MK02` для MM поставщиков; в S/4HANA использовать `BP`)
2. Выбрать поставщика (например, `100001`), балансовую единицу (`1000`)
3. Перейти в раздел: **Платёжные операции**
4. Поле: `Дополнение к способу оплаты` (`UZAWE`) → ввести `01`
5. Сохранить

> Затронутая таблица: `LFB1` — поле `UZAWE`.

**Массовое обновление через LSMW:**

1. Транзакция: `LSMW`
2. Создать новый проект/подпроект/объект
3. Метод записи: BDC-запись транзакции `XK02`
4. Записать следующие шаги:
   - Ввести номер поставщика и балансовую единицу
   - Перейти в раздел платёжных операций
   - Изменить поле `UZAWE` на `01`
   - Сохранить
5. Подготовить входной файл (CSV/Excel) со столбцами: номер поставщика, балансовая единица, значение дополнения (`01`)
6. Выполнить маппинг исходных полей на поля записи
7. Выполнить: Чтение данных → Конвертация → Создание сессии пакетного ввода → Запуск сессии

**Массовое обновление через прямое изменение таблицы (только в экстренных случаях):**

Транзакция `SE16N` или пользовательская ABAP-программа для прямого обновления `LFB1-UZAWE`. Не рекомендуется для продуктива из-за отсутствия создания документов об изменениях. Использовать только если LSMW/BDC неприменимы, дополняя обновление ручными записями документов об изменениях.

**Критерии выборки поставщиков для пилотной группы:**

Выбрать 5-10 поставщиков на способ оплаты, отвечающих ВСЕМ следующим критериям:

- Низкая частота платежей (еженедельно или ежемесячно, не ежедневно) — ограничивает масштаб последствий при сбое формата
- Небольшие суммы платежей — снижает финансовые риски
- Только внутренние платежи (изначально) — избегает сложности трансграничных платежей
- Гарантированно хорошие мастер-данные: заполнен IBAN, структурированный адрес, банковские реквизиты проверены
- Кооперативное взаимодействие — поставщик подтвердит получение и быстро сообщит о проблемах
- Распределены по разным домашним банкам (если их несколько) — проверяет приём со стороны банка

#### A.2.4 Постепенное расширение (10% → 50% → 100%)

| Волна | Популяция поставщиков | Метод | Период валидации |
|---|---|---|---|
| Пилот (Этап 1) | 5-10 на способ оплаты (50-100 итого) | Ручной `XK02` или небольшой пакет LSMW | 2-4 недели |
| Расширение (Этап 2) | ~50% поставщиков на способ оплаты | Пакетный прогон LSMW | 2-3 недели |
| Полная миграция (Этап 3) | Оставшиеся 50% | Пакетный прогон LSMW | 1-2 недели |
| Очистка (Этап 4) | 0% на дополнении (все на базовом) | Прямая замена формата в базовом ЗО, затем очистка всех дополнений | 1 неделя |

**Процедура расширения для каждой волны:**

1. Подготовить список поставщиков (CSV) для волны: номер поставщика, балансовая единица, дополнение = `01`
2. Сначала выполнить LSMW в QAS → проверить через `SE16` в `LFB1`, что `UZAWE` установлен правильно
3. Перенести / повторно выполнить LSMW в PRD
4. Выполнить следующий прогон F110 (сначала только предложение) → проверить в журнале предложения, что поставщики с дополнением идентифицированы
5. Проверить предварительный просмотр DME через `F110` → Среда → Платёжный носитель → Администрирование DME
6. Если вывод DME корректен → выполнить платёжный прогон
7. Отслеживать банковские подтверждения для DME-файлов в новом формате
8. Подождать окончания периода валидации перед следующей волной

#### A.2.5 Процедура отката на каждом этапе

Откат одинаков на каждом этапе и может быть выполнен на трёх уровнях детализации:

**Откат по отдельному поставщику:**

1. Транзакция: `XK02` → поставщик → балансовая единица → Платёжные операции
2. Очистить поле `UZAWE` (установить в пустое значение)
3. Сохранить
4. Следующий прогон F110: данный поставщик возвращается к формату базового способа оплаты (старый формат)

**Откат по волне (массовый):**

1. Подготовить входной файл LSMW с тем же списком поставщиков, использованным для волны, но с `UZAWE` = пусто
2. Выполнить LSMW
3. Все поставщики волны возвращаются к старому формату

**Полный откат (все этапы):**

1. Запустить LSMW для ВСЕХ поставщиков, у которых `UZAWE` = `01`, устанавливая значение в пусто
2. Проверить через `SE16` → `LFB1` → фильтр `UZAWE = '01'` → количество должно быть 0
3. Изменений FBZP не требуется — конфигурация дополнения (`T042D`) может оставаться на месте (неактивной) или быть удалена позже

**Время отката:** Откат может быть выполнен в любое время, включая середину дня между прогонами F110. Изменение вступает в силу на следующем платёжном прогоне, обрабатывающем затронутого поставщика.

| Проверка | Частота | Инструмент | Действие при сбое |
|---|---|---|---|
| DME-файл сгенерирован без ошибок | Каждый прогон F110 | `FPY1`, `SM37` | Исследовать журнал ошибок; откатить затронутых поставщиков при критической ситуации |
| Подтверждение принятия банком | В течение 24ч после передачи DME | Банковский портал / pain.002 | Связаться с банком; откатить при систематическом отклонении |
| Сверка сумм платежей | Ежедневно | `REGUH` vs. банковская выписка | Исследовать расхождение; приостановить расширение при выявлении закономерности |
| Жалобы поставщиков | Постоянно | Служба поддержки / email | Расследовать по каждому поставщику; откатить отдельного поставщика при необходимости |
| Время выполнения F110 в рамках SLA | Каждый прогон F110 | Длительность задания в `SM37` | Оптимизировать вариант выборки или разбить прогон при риске превышения времени |

#### Критерии валидации перед переходом к Этапу 3

- [ ] Не менее 2 полных циклов F110 завершены с 50% популяцией поставщиков на новом формате
- [ ] Уровень принятия банком: 100% (ноль отклонений)
- [ ] Расхождений при сверке платежей нет
- [ ] Время выполнения F110 в приемлемых рамках SLA (деградация не более 15%)
- [ ] Трансграничные платежи (если включены) обработаны корректно
- [ ] Систематических жалоб поставщиков, связанных с изменением формата, нет

#### Процедура отката

1. Подготовить файл реверсирования LSMW: все поставщики Этапа 2 с `UZAWE` = пусто
2. Выполнить LSMW
3. Проверить количество
4. Пилотные поставщики Этапа 1 могут остаться на новом формате (они проверены) или также быть отозваны — в зависимости от характера проблемы
5. Время отката: < 2 часов включая проверку

### Этап 3: Полная миграция (100% поставщиков)

**Длительность:** 1-2 недели  
**Трудозатраты:** 6-8 человеко-дней

#### Действия по настройке

| # | Действие | Транзакция | Детали |
|---|---|---|---|
| 1 | Подготовить финальный список поставщиков | `SE16` → `LFB1` фильтр `UZAWE <> '01'` AND `ZWELS` = соответствующий ЗО | Все оставшиеся поставщики, ещё не перешедшие на дополнение `01` |
| 2 | Выполнить массовое обновление LSMW | `LSMW` | Установить `LFB1-UZAWE` = `01` для всех оставшихся поставщиков |
| 3 | Проверить 100% охват | `SE16` → `LFB1` | Для каждого ЗО и балансовой единицы: все поставщики с данным ЗО должны иметь `UZAWE` = `01`. Сверить количество с общей базой поставщиков по ЗО |
| 4 | Выполнить прогоны F110 | `F110` | Стандартные платёжные прогоны. Поток DME в старом формате теперь должен быть пустым или почти пустым |
| 5 | Подтвердить, что поток старого формата пуст | `FDTA`, `FPY1` | Убедиться, что DME-файлы в старом формате не генерируются. Если таковые есть — расследовать: вероятно, пропущенный поставщик или новый поставщик, созданный без дополнения |
| 6 | Вести мониторинг в течение 1-2 полных платёжных циклов | Весь мониторинг из Этапа 2 | Подтвердить стабильность 100% обработки в новом формате |

#### Критерии валидации перед переходом к Этапу 4

- [ ] 100% поставщиков по всем ЗО и балансовым единицам имеют `UZAWE` = `01`
- [ ] Не менее 2 полных циклов F110 со 100% выводом в новом формате
- [ ] Ноль DME-файлов в старом формате
- [ ] Ноль банковских отклонений
- [ ] Полная сверка пройдена
- [ ] Одобрение со стороны казначейства / операций КЗ

#### Процедура отката

Аналогично Этапу 2, но в большем масштабе. Полный откат всех поставщиков к пустому `UZAWE`:

1. Подготовить файл реверсирования LSMW: ВСЕ поставщики, `UZAWE` = пусто
2. Выполнить LSMW
3. Время отката: < 4 часов включая проверку для крупных баз поставщиков

> **Важно:** После проверки Этапа 3 окно отката начинает закрываться. После выполнения Этапа 4 (прямая замена формата) откат требует отмены изменения конфигурации FBZP, что значительно сложнее.

### Этап 4: Очистка

**Длительность:** 1-2 недели  
**Трудозатраты:** 5-7 человеко-дней

#### Действия по настройке

| # | Действие | Транзакция | Детали |
|---|---|---|---|
| 1 | Заменить формат базового способа оплаты | `FBZP` → ЗО в балансовой единице | Для каждого ЗО в каждой балансовой единице: изменить базовый формат (без дополнения) со старого на новый. Пример: ЗО `T` в балансовой единице `1000` → изменить «Формат платёжного носителя» с `RFFOD__T` на `SEPA_CT` |
| 2 | Удалить назначение формата дополнения | `FBZP` → ЗО в балансовой единице → Дополнения; `OBPM` | Удалить назначение формата из дополнения `01` (или полностью удалить определение дополнения из `T042D`). Теперь базовый формат ЗО обрабатывает всё |
| 3 | Очистить поля дополнений у поставщиков | `LSMW` | Массово обновить ВСЕХ поставщиков: установить `LFB1-UZAWE` = пусто. Дополнение больше не нужно, так как базовый ЗО теперь использует новый формат |
| 4 | Проверить обработку базового формата | `F110` | Выполнить платёжный прогон. Убедиться, что все поставщики производят вывод DME в новом формате через базовый ЗО (без участия дополнений) |
| 5 | Удалить артефакты старого формата | `DMEE` (опционально) | Пометить старые деревья форматов как неактивные или удалить, если они больше не используются ни в одном ЗО ни в одной балансовой единице. Удалить старые классические программы из назначения на уровне страны в FBZP, если они полностью заменены PMW |
| 6 | Очистить варианты выборки | `OBPM4` | Удалить варианты выборки, специфичные для дополнений, которые больше не нужны. Сохранить базовые варианты для нового формата |
| 7 | Обновить документацию | Внешнее | Обновить: документацию по конфигурации FBZP, инструкции по работе с F110, документы со спецификациями платёжных форматов, записи банковской коммуникации |
| 8 | Архивировать аудиторский след | Внешнее | Экспортировать журнал миграции: по поставщику, ЗО, балансовой единице — когда было назначено дополнение, когда первые платежи выполнялись в новом формате, когда дополнение было очищено. Хранить в соответствии с политикой хранения для аудиторских целей |
| 9 | Деактивировать BAdI (если использовался) | `SE19` | Если в Этапе 0 был реализован логирующий BAdI — деактивировать или удалить его. Перенести деактивацию |
| 10 | Обновить процесс создания новых поставщиков | Организационное | Обновить процедуры создания мастер-данных поставщиков: новые поставщики НЕ должны получать значение дополнения (базовый формат ЗО теперь является новым форматом). Обновить автоматизированные интерфейсы создания поставщиков (IDoc, обёртки BAPI) |

#### Критерии валидации — Миграция завершена

- [ ] Все ЗО во всех балансовых единицах: базовый формат = новый формат (проверено в `T042Z`)
- [ ] Все поставщики: `UZAWE` = пусто (проверено в `LFB1`)
- [ ] Конфигурация дополнений удалена или деактивирована в `T042D`
- [ ] Не менее 2 полных циклов F110 завершено после очистки только с базовым форматом
- [ ] Деревья старого формата деактивированы / заархивированы
- [ ] Документация обновлена
- [ ] Официальное одобрение от всех заинтересованных сторон (Казначейство, Операции КЗ, ИТ, Аудит)

#### Процедура отката

Откат после Этапа 4 сложнее, поскольку формат базового способа оплаты был изменён:

1. Повторно перенести старую конфигурацию FBZP (восстановить старое назначение формата для базового ЗО) из сохранённого переноса или настроить вручную
2. Именно поэтому Этап 4 следует выполнять только после проверки Этапа 3 в течение не менее 2 полных платёжных циклов
3. Рассмотреть возможность оставить старое дерево форматов в системе (неактивным) на 3-6 месяцев после миграции в качестве аварийного запасного варианта

## Часть D — Риски и меры по их снижению

### Риск 1: Готовность банка к принятию нового формата

**Описание:** Домашний банк (или банки) может быть не готов принять новую версию формата (например, pain.001.001.09) к моменту запуска пилота. Банки могут иметь собственные сроки тестирования и сертификации, не совпадающие с расписанием миграции SAP.

**Вероятность:** Средняя  
**Влияние:** Высокое — платежи отклонены банком, что приводит к задержкам и возможным штрафам за просрочку.

**Меры по снижению:**

1. Вступить в контакт с каждым домашним банком за 8-12 недель до Этапа 1. Получить письменное подтверждение принятия формата и поддерживаемой конкретной версии/схемы.
2. Использовать тестовый канал подачи файлов банка (если доступен) для отправки образцов DME-файлов из QAS до запуска в PRD.
3. Если банк не готов — задержать Этап 1 для данного домашнего банка / балансовой единицы, продолжая работу с другими.
4. Сохранять старый формат в качестве запасного варианта (через откат дополнений) до получения подтверждения от банка.

### Риск 2: Пробелы в валидации DME / IDoc файлов

**Описание:** В новом формате могут быть проблемы с маппингом полей, не выявленные при тестировании в DEV/QAS: отсутствующие обязательные элементы (например, структурированный адрес, если банк его ожидает), неверные URI пространства имён, неправильная кодировка символов, неверная структура XML или отсутствие UETR.

**Вероятность:** Средняя  
**Влияние:** Высокое — отклонение банком или неверная маршрутизация платежей.

**Меры по снижению:**

1. Провести пофилдное сравнение старого и нового вывода DME для одного и того же платежа в QAS. Задокументировать каждое различие.
2. Валидировать DME-файлы в новом формате по официальной XML-схеме (XSD) с помощью внешнего XML-валидатора перед отправкой в банк.
3. Для ISO 20022: валидировать по отчёту об определении сообщений ISO 20022 (MDR) для целевой версии.
4. Проверить SAP Note 2665657 (примечания к формату ISO 20022) и Note 1682833 (исправления формата SEPA) на предмет известных проблем с маппингом полей.
5. Пилотный этап (Этап 1) с 5-10 поставщиками специально тестирует эти пробелы при малом объёме и малом риске.

### Риск 3: Несоответствия мастер-данных поставщиков

**Описание:** Новый формат (особенно ISO 20022) имеет более строгие требования к данным, чем устаревшие форматы: IBAN обязателен (не только номер банковского счёта), структурированные адреса требуют отдельных полей для улицы/города/почтового индекса/страны (а не одной строки адреса), BIC может потребоваться для трансграничных платежей. Поставщики с неполными мастер-данными будут вызывать ошибки генерации формата.

**Вероятность:** Высокая — почти в каждой SAP-системе есть устаревшие поставщики с неполными данными.  
**Влияние:** Среднее — платежи отдельных поставщиков завершаются неудачей; ошибки генерации формата в `FPY1`.

**Меры по снижению:**

1. Запустить отчёт о качестве мастер-данных в Этапе 0: запросить `LFB1` + `LFBK` (банковские реквизиты) + `LFA1` (адрес) для всех поставщиков, назначенных на 10 способов оплаты.
2. Отметить поставщиков с: отсутствующим IBAN, отсутствующим BIC (для трансграничных платежей), неструктурированным адресом (одна строка `STRAS` вместо структурированных полей), отсутствующим кодом страны.
3. Устранить критические проблемы до Этапа 1. Для крупных популяций в первую очередь исправить пилотных поставщиков, затем пакетно исправить оставшихся до Этапа 2.
4. Рассмотреть возможность реализации валидации в варианте выборки PMW или BAdI предобработки, который отклоняет поставщиков с неполными данными, а не генерирует некорректный DME-файл.
5. Для мандата структурированного адреса с ноября 2026: провести аудит `LFA1-STRAS` vs. `LFA1-STRAS` + `ORT01` + `PSTLZ` + `LAND1` и конвертировать в структурированный формат.

### Риск 4: Поле UZAWE уже используется

**Описание:** Поле дополнения к способу оплаты (`LFB1-UZAWE`) может уже быть заполнено для некоторых или всех поставщиков и использоваться в других целях (например, для дифференциации форм чеков, маршрутизации конкретных платёжных авизо или поддержки предыдущей миграции формата, которая так и не была очищена).

**Вероятность:** Низкая-Средняя (зависит от истории клиента).  
**Влияние:** Высокое — перезапись существующих значений дополнений нарушает существующую логику обработки платежей.

**Меры по снижению:**

1. **Аудит в Этапе 0:** Выполнить `SE16` → `LFB1` с фильтром `UZAWE <> ''` для выявления всех поставщиков с существующими значениями дополнений. Проанализировать, что эти значения делают.
2. **Если дополнения активно используются:** Выбрать другое значение дополнения (например, `N1` вместо `01`), которое не конфликтует. Создать новое дополнение в `T042D` с этим значением.
3. **Если дополнения устаревшие (осиротевшие):** Сначала очистить осиротевшие значения, затем использовать `01` по плану.
4. **Если дополнения критически важны и полностью используются:** Перейти к подходу через BAdI (A.1.3) в качестве основного механизма вместо дополнений.

### Риск 5: Влияние на полномочия и роли

**Описание:** У ответственных за ведение мастер-данных поставщиков может не быть полномочий для изменения поля `UZAWE`. Операторам F110 могут потребоваться дополнительные полномочия для обработки платежей в новом формате. Может потребоваться корректировка объектов полномочий `F_LFB1_BUK` (данные поставщика по балансовой единице) и `F_REGU_BUK` (полномочия на платёжный прогон).

**Вероятность:** Средняя  
**Влияние:** Среднее — задержки в обновлениях мастер-данных поставщиков или выполнении платёжных прогонов.

**Меры по снижению:**

1. В Этапе 0 убедиться, что назначенная команда миграции (исполнитель LSMW, ответственные за мастер-данные поставщиков) имеет полномочия для изменения `LFB1-UZAWE` во всех целевых балансовых единицах.
2. Убедиться, что операторы F110 имеют полномочия для выполнения нового формата PMW (объект полномочий `F_PMTFMT`).
3. При необходимости создать временную роль полномочий для команды миграции с определённой датой истечения.
4. Протестировать полномочия в QAS до запуска Этапа 1.

### Риск 6: Конфликты при планировании платёжных прогонов

**Описание:** Этапы 1-3 могут совпасть с периодами высокого объёма платежей (конец месяца, квартала, года), что увеличивает риск проблем и сокращает время для валидации и устранения неполадок.

**Вероятность:** Средняя  
**Влияние:** Среднее — давление с пропуском шагов валидации; больший масштаб последствий при возникновении проблем в пиковые периоды.

**Меры по снижению:**

1. **Запланировать пилот Этапа 1 на неделю с низким объёмом** (середина месяца, не конец месяца или квартала).
2. **Не расширять волны (Этапы 2, 3) в период закрытия месяца** — заморозить на последние 5 рабочих дней каждого месяца.
3. **Этап 4 (прямая замена) запланировать на первую неделю после окончания квартала**, когда следующий пик наиболее далёк.
4. Поддерживать чёткий «календарь заморозки миграции», разделённый с операциями КЗ.

### Риск 7: Несоответствие версий ISO 20022

**Описание:** Целевая версия формата в дереве форматов SAP может не совпадать с версией, требуемой банком или клиринговой организацией. Например, SAP может поставляться с pain.001.001.03 (версия 2009), тогда как банк требует pain.001.001.09 (версия 2019). Или SAP-система может быть на более старом пакете поддержки, не включающем деревья форматов версии 2019.

**Вероятность:** Средняя (особенно в более старых системах ECC).  
**Влияние:** Высокое — банк отклоняет весь DME-файл.

**Меры по снижению:**

1. Подтвердить точную требуемую версию формата у каждого домашнего банка (pain.001.001.03 vs. .09). Задокументировать в плане миграции.
2. Проверить уровень пакета поддержки SAP и доступные деревья форматов в `DMEE`. Для версий 2019 (pain.001.001.09) SAP Note 2816629 и последующие ноты предоставляют обновлённые деревья форматов.
3. Если требуемая версия недоступна — применить соответствующий пакет поддержки SAP или импортировать дерево форматов через вложение к ноте.
4. Для пользовательских деревьев форматов (`Z_*`) убедиться, что атрибуты XML-пространства имён и версии совпадают с целевой версией.
5. Валидировать сгенерированный XML по официальному XSD для целевой версии перед отправкой в банк.

- [ ] Проверить, что индикатор дополнения (`T042C-XZAWE`) активен для всех ЗО во всех странах

- [ ] Связаться с каждым домашним банком — подтвердить принятие нового формата и версию
- [ ] Получить тестовый канал подачи файлов банка (если доступен)
- [ ] Проверить роли полномочий: ведение `UZAWE`, выполнение F110, полномочия на формат PMW

- [ ] Выполнить сквозное тестирование в QAS: то же, что в DEV, с подачей тестовых файлов в банк при наличии возможности
- [ ] Сравнить вывод DME в старом и новом форматах поле за полем — задокументировать все различия
- [ ] Устранить все расхождения при сравнении DME
- [ ] Перенести всю конфигурацию в PRD (FBZP, OBPM, OBPM4, DMEE)
- [ ] Подготовить список пилотных поставщиков (5-10 на ЗО, отвечающих критериям из раздела A.2.3)

- [ ] Протестировать LSMW в QAS с выборкой поставщиков
- [ ] Установить календарь заморозки миграции (никаких изменений в конце месяца/квартала)
- [ ] Запланировать дату начала Этапа 1 (неделя с низким объёмом)

- [ ] Назначить дополнение `01` пилотным поставщикам через `XK02` или LSMW (50-100 поставщиков итого)
- [ ] Проверить назначение в `SE16` → `LFB1` → фильтр `UZAWE = '01'` — количество совпадает с пилотным списком
- [ ] Выполнить прогон предложения F110 — проверить журнал предложения для пилотных поставщиков
- [ ] Просмотреть предварительный просмотр DME в `F110` → Среда → Платёжный носитель

- [ ] Выполнить платёжный прогон F110
- [ ] Контролировать журнал платёжного носителя в `FPY1` — ноль ошибок
- [ ] Передать DME-файл в новом формате в банк
- [ ] Подтвердить принятие банком (нет отклонений в течение 24ч)

- [ ] Убедиться, что поставщики со старым форматом продолжают обрабатываться нормально (нет регрессий)
- [ ] Завершить не менее 2 полных циклов F110 с пилотной группой
- [ ] Задокументировать результаты по поставщикам: идентификатор прогона, использованный формат, принятие банком
- [ ] Получить одобрение от операций КЗ для расширения Этапа 2

### Этап 2: Расширение (50%)

- [ ] Подготовить список поставщиков для расширения (~50% оставшихся поставщиков на ЗО)
- [ ] Выполнить LSMW в QAS — проверить корректное назначение `UZAWE`
- [ ] Выполнить LSMW в PRD
- [ ] Проверить в `SE16` → `LFB1` — ~50% базы поставщиков на ЗО имеет `UZAWE = '01'`
- [ ] Выполнить прогоны F110 — контролировать увеличение объёма DME в новом формате
- [ ] Проверить пропорциональность количества записей и итогов DME-файла охвату поставщиков
- [ ] Контролировать обработку в банке для больших объёмов файлов
- [ ] Если включены международные поставщики: проверить трансграничные XML-элементы
- [ ] Контролировать время выполнения F110 — убедиться в отсутствии деградации SLA более 15%
- [ ] Ежедневная сверка в течение 2 недель: `REGUH`/`REGUP` vs. банковские выписки
- [ ] Контролировать жалобы поставщиков — расследовать проблемы, связанные с форматом
- [ ] Завершить не менее 2 полных циклов F110 при 50% охвате
- [ ] Получить одобрение для Этапа 3

### Этап 3: Полная миграция (100%)

- [ ] Подготовить финальный список поставщиков: все оставшиеся поставщики без дополнения
- [ ] Выполнить LSMW в PRD — установить `UZAWE = '01'` для всех оставшихся поставщиков
- [ ] Проверить 100% охват: `SE16` → `LFB1` — все поставщики с данным ЗО имеют `UZAWE = '01'`
- [ ] Выполнить прогоны F110
- [ ] Подтвердить, что поток DME в старом формате пуст (ноль сгенерированных файлов в старом формате)
- [ ] Исследовать любой оставшийся вывод в старом формате — пропущенные поставщики, новые поставщики
- [ ] Установить процесс создания новых поставщиков: дополнение не нужно (будет обработано в Этапе 4)
- [ ] Завершить не менее 2 полных циклов F110 при 100% охвате новым форматом
- [ ] Полная сверка — ноль расхождений
- [ ] Получить одобрение от Казначейства, Операций КЗ и ИТ для Этапа 4

### Этап 4: Очистка

- [ ] Заменить базовый формат ЗО: `FBZP` → ЗО в балансовой единице → изменить формат со старого на новый для каждого ЗО / балансовой единицы
- [ ] Удалить назначение формата дополнения из `T042D` (или удалить дополнение `01`)
- [ ] Массово очистить `LFB1-UZAWE` у всех поставщиков через LSMW (установить в пусто)
- [ ] Проверить: `SE16` → `LFB1` → фильтр `UZAWE = '01'` → количество = 0
- [ ] Выполнить прогон F110 с базовым форматом ЗО (без дополнений) — проверить, что вывод DME в новом формате
- [ ] Проверить 2 полных цикла F110 после очистки
- [ ] Деактивировать / удалить реализацию BAdI (если использовалась для логирования)
- [ ] Удалить специфичные для дополнений варианты выборки PMW из `OBPM4`
- [ ] Деактивировать деревья старого формата в `DMEE` (хранить 3-6 месяцев как аварийный запасной вариант)
- [ ] Обновить процедуры создания мастер-данных поставщиков: новые поставщики используют базовый ЗО (новый формат по умолчанию)
- [ ] Обновить инструкцию по работе с F110 / операционные процедуры
- [ ] Обновить документацию по конфигурации FBZP
- [ ] Архивировать аудиторский след миграции (по поставщику: даты назначения дополнений, даты изменения формата, идентификаторы прогонов)
- [ ] Официальное одобрение: миграция завершена — от Казначейства, Операций КЗ, ИТ, Внутреннего аудита
- [ ] Запланировать удаление деревьев старого формата: через 6 месяцев после миграции
