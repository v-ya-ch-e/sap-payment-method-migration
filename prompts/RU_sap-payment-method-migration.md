# Промпт: SAP Способы оплаты — Стратегия поэтапной миграции форматов

> **Применение:** Передайте этот промпт (вместе с `CLAUDE.md` в качестве контекста) языковой модели, чтобы получить детальный и практически применимый план миграции платёжных форматов SAP в продуктивной среде.

---

## Промпт

```text
Вы — старший SAP FI/AP консультант с опытом более 15 лет в области обработки платежей,
настройки FBZP, Payment Medium Workbench (PMW) и крупных продуктивных миграций.

## Ситуация

Клиент имеет следующий SAP-ландшафт:

- Несколько балансовых единиц (каждая со своей конфигурацией FBZP).
- 10 различных способов оплаты, активно используемых в каждой балансовой единице
  (например, T = банковский перевод, C = чек, U = срочный перевод и т.д.).
- Каждый способ оплаты в настоящее время использует СТАРЫЙ платёжный формат
  (либо классическую программу RFFO*, либо устаревшее дерево форматов PMW).
- Цель — мигрировать ВСЕ способы оплаты во ВСЕХ балансовых единицах на НОВЫЙ
  платёжный формат (например, ISO 20022 pain.001/pain.002, CGI XML или новое
  пользовательское дерево форматов PMW).
- Система является продуктивной средой SAP ECC или S/4HANA с ежедневными
  платёжными прогонами через F110.
- Каждому способу оплаты назначены сотни поставщиков.

## Ограничения

1. **Риск исчерпания символов:** Коды способов оплаты однобуквенные (A-Z, 0-9).
   Создание дублирующего способа оплаты для каждого существующего (10 старых + 10 новых = 20)
   может исчерпать доступные символы, особенно если некоторые уже заняты.
   Кроме того, дублирование всей конфигурации (настройки FBZP, определение банка,
   домашние банки, формы, права доступа) для каждого способа оплаты является
   огромной нагрузкой на обслуживание.

2. **Риск «большого взрыва»:** Простая замена формата в существующем способе оплаты
   (прямая замена) немедленно затрагивает каждого поставщика и каждый платёжный прогон,
   использующий этот метод. Если новый формат имеет проблемы (отклонение банком, ошибки
   маппинга полей, недостающие данные), весь платёжный процесс нарушается без
   возможности простого отката.

3. **Безопасность продуктива:** Клиент требует:
   - Поэтапного развёртывания (начало с малой пилотной группы, постепенное расширение).
   - Возможности отката на старый формат в любой момент без перенастройки.
   - Минимального нарушения ежедневных операций F110.
   - Отсутствия изменений параметров прогона F110, вариантов выборки или рабочих
     процессов пользователей (по возможности).

4. **Прослеживаемость:** Каждый шаг миграции должен быть прослеживаем. Клиент должен
   иметь возможность продемонстрировать, какие поставщики использовали какой формат
   в какую дату.

## Ваша задача

Предоставьте **детальную пошаговую стратегию миграции**, охватывающую следующее:

### Часть A — Рекомендуемый подход

1. Оцените следующие механизмы миграции и рекомендуйте ЛУЧШИЙ основной подход
   (или их комбинацию). Для каждого объясните, почему он подходит или не подходит:
   - **Дополнения к способу оплаты (Zahlwegergänzung)** — использование 2-символьного поля
     дополнения (UZAWE / T042D) для маршрутизации пилотных поставщиков на новый формат
     при сохранении базового способа оплаты на старом формате.
   - **Переопределение формата на уровне домашнего банка / идентификатора счёта** —
     создание параллельного идентификатора счёта с новым форматом и корректировка
     приоритетов в определении банка.
   - **Динамическое переключение форматов через BAdI** — реализация FI_PAYMEDIUM или
     аналогичного BAdI для выбора формата во время выполнения на основе списка пилотных
     поставщиков / Z-таблицы / диапазона дат.
   - **Временный дублирующий способ оплаты** — создание одного временного кода способа
     оплаты для каждого исходного, используемого только в период миграции, затем
     выводимого из эксплуатации.
   - **Параллельные прогоны F110** — отдельные платёжные прогоны для поставщиков
     со старым и новым форматами.
   - **Любой другой стандартный механизм SAP**, который мог быть упущен.

2. Для рекомендуемого подхода предоставьте:
   - Точные шаги настройки FBZP (какие разделы изменить, какие поля).
   - Как разрешение формата работает во время выполнения (трассировка логики от F110 →
     SAPFPAYM → выбор формата).
   - Как управляется назначение поставщиков в пилотную группу.
   - Как постепенно расширять пилот (10% → 50% → 100%).
   - Как работает откат на каждом этапе.

### Часть B — Этапы миграции

Предоставьте поэтапный план миграции с конкретными шагами:

- **Этап 0: Подготовка** — что настроить, протестировать в DEV/QAS, согласовать с банками.
- **Этап 1: Пилот (5-10 поставщиков на способ оплаты)** — запуск в продуктив с малой
  группой, критерии валидации, сравнение DME-файлов (старый и новый форматы).
- **Этап 2: Расширение (50%)** — масштабирование до половины поставщиков, план мониторинга.
- **Этап 3: Полная миграция (100%)** — перевод оставшихся поставщиков, финальная замена формата.
- **Этап 4: Очистка** — удаление временной конфигурации, обновление документации.

Для каждого этапа укажите:
- Точные действия по настройке (транзакции, таблицы, поля).
- Критерии валидации/приёмки перед переходом к следующему этапу.
- Процедуру отката при обнаружении проблем.
- Ожидаемые трудозатраты (человеко-дни) для команды из 2 SAP-консультантов.

### Часть C — Сравнительная таблица

Предоставьте сравнительную таблицу всех подходов со следующими столбцами:
- Название подхода
- Гранулярность на уровне поставщика (да/нет)
- Требует изменений мастер-данных поставщика (да/нет)
- Требует пользовательского кода (да/нет)
- Расходует символы способов оплаты (да/нет)
- Сложность отката (низкая/средняя/высокая)
- Операционная сложность (низкая/средняя/высокая)
- Рекомендуется для данного сценария (да/нет + краткое обоснование)

### Часть D — Риски и меры по их снижению

Перечислите 5-7 основных рисков, характерных для данной миграции, и способы снижения
каждого из них. Включите:
- Готовность банка к принятию нового формата
- Пробелы в валидации IDoc / DME-файлов
- Несоответствия мастер-данных
- Влияние на права доступа / роли
- Конфликты при планировании платёжных прогонов

### Часть E — Чеклист

Предоставьте готовый к использованию чеклист (в формате markdown-чекбоксов), которому
команда может следовать на каждом этапе миграции.

## Формат вывода

- Включайте конкретные коды транзакций SAP, имена таблиц и имена полей.
```

## Как использовать этот промпт

1. **Сначала предоставьте контекст:** Вставьте содержимое `CLAUDE.md` (или прикрепите его)
   перед этим промптом, чтобы языковая модель имела необходимые предметные знания.

2. **Настройте переменные:** Перед отправкой замените или уточните:
   - Точное **название старого формата** (например, `RFFOD__T`, конкретный ключ формата PMW).
   - Точное **название нового формата** (например, `SEPA_CT`, `CGI_XML_V3`, ваш пользовательский Z-формат).
   - **Количество балансовых единиц**.
   - Используете ли вы **ECC** или **S/4HANA** (некоторые BAdI различаются).
   - Любые **страновые ограничения** (например, немецкий SEPA, американский ACH, швейцарский SIC).

3. **Итерируйте:** После первого ответа задавайте уточняющие вопросы:
   - «Покажите точные экраны FBZP / пошаговую настройку полей для Этапа 1.»
   - «Создайте шаги записи LSMW/BDC для массового обновления поля дополнения в мастер-данных поставщиков.»
   - «Какие SAP OSS-ноты актуальны для миграции форматов PMW?»
   - «Как это будет отличаться при использовании подхода через дополнения к способу оплаты
     vs. подхода через BAdI — покажите оба варианта рядом.»

4. **Сверяйтесь с документацией SAP:** Перепроверяйте вывод модели по:
   - SAP Help Portal: [Payment Medium Workbench](https://help.sap.com)
   - SAP Note 1541877 (руководство по миграции форматов PMW)
   - SAP Note 986210 (дополнения к способам оплаты)
